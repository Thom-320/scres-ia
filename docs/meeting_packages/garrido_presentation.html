Alignment Fixes Applied

4 discrepancies found and corrected in this audit cycle

Fix 1: Op4 Transit Delay

Op3 (reception, 24h) and Op4 (transport, 24h) were merged into one 24h step. Now correctly modeled as two separate operations totaling ~48h pipeline delay.

supply_chain.py — _op3_wdc_dispatch()

Fix 2: Shift ↔ Batch Auto-Coupling

Table 6.20: S=3 → batch_size=7,000. Previously static at 5,000. Now auto-updates via CAPACITY_BY_SHIFTS when assembly_shifts changes.

supply_chain.py — step() + live batch_size read

Fix 3: Fill Rate (Eq 5.1)

Was order-count based. Now quantity-based per thesis: FR = 1 − (B + U) / D using the backorder queue state.

supply_chain.py — _fill_rate() + _backorder_rate()

Fix 4: Backorder Queue (Sec 6.5.4)

Implemented Garrido's 60-order capped queue with SPT scheduling, contingent demand priority, and Ut (unattended) tracking.

supply_chain.py — _enqueue_backorder(), _serve_pending_backorders()

Scalar vs Vector Representation

David's concern: "the implementation uses scalars where the thesis defines vectors"

The thesis defines Q at Op3 as a 12-element vector: {15,500 rm1, ..., 15,500 rm12}.

Our implementation stores Q as a scalar per material (15,500) and multiplies by NUM_RAW_MATERIALS = 12 when dispatching.

Mathematically equivalent — all 12 raw materials are identical in the thesis (same Q, PT, risk distributions).

For DKANA, the state constraint context already provides vector-structured data:

DKANA Vectors Provided

  • 7-dim inventory by buffer node
  • 7-dim backorder rate by node
  • 13-dim disruption fraction by operation
  • 3-dim backorder queue state
  • 20-dim observation (v3)
  • 9-dim control context

Total state constraint: 45 dimensions
Row matrix for DKANA: 65 × 3

What we have now (step-level)

ReT is computed per RL step (168h window) using aggregated fill rate and disruption fraction. This is a publishable approximation (assumptions A1–A4 documented).

4-case classification works at step level:

  • Case 1: fill_rate_only (no disruption)
  • Case 2: autotomy (high FR despite disruption)
  • Case 3: recovery (degraded FR + disruption)
  • Case 4: non-recovery (severe collapse)

What's missing (order-level)

The thesis computes APj, RPj, DPj per order j by tracking which risk event(s) overlap each order's lifecycle (OPTj → OATj). This requires:

  • Linking each order to concurrent risk events
  • Computing disruption overlap per order
  • Classifying AP/RP/DP per order (not per step)
  • Producing full SDM matrix for validation

Question for Garrido: For the RL publication, is the step-level ReT approximation sufficient with the documented assumptions? Or do we need the full order-level SDM to validate the simulation before running RL experiments?
